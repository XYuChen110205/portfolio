## 1. 平陆运河 · 世纪工程数字叙事

### 1.1 项目背景与意义
平陆运河是新中国成立以来首条通江达海的运河工程，全长约134.2公里，总投资约727亿元。为了让公众更好地理解这一世纪工程的意义，项目采用"数字叙事"（Digital Storytelling）理念，通过Web前端技术将枯燥的工程数据转化为生动的视觉体验。系统以沉浸式滚动叙事为核心，融合WebGL 3D渲染和GSAP高性能动画，实现桌面端和移动端的全平台覆盖。

### 1.2 系统架构
```
┌─────────────────────────────────────────────────────┐
│                    展示层 (Presentation)              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐ │
│  │首页·总览 │ │详细介绍  │ │运河记忆  │ │关于页面│ │
│  │          │ │          │ │(时间线)  │ │        │ │
│  └──────────┘ └──────────┘ └──────────┘ └────────┘ │
├─────────────────────────────────────────────────────┤
│                  渲染引擎层 (Rendering)              │
│  ┌──────────────────┐  ┌─────────────────────────┐  │
│  │ Three.js WebGL   │  │ GSAP 3 ScrollTrigger    │  │
│  │ (3D粒子隧道背景) │  │ (滚动驱动动画编排)       │  │
│  └──────────────────┘  └─────────────────────────┘  │
├─────────────────────────────────────────────────────┤
│                  资源层 (Assets)                     │
│  ┌──────────┐ ┌──────────┐ ┌────────────────────┐  │
│  │3D粒子系统│ │图片素材  │ │Font Awesome + 字体 │  │
│  └──────────┘ └──────────┘ └────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### 1.3 核心业务流程
```
用户访问 → 首页加载 → Three.js粒子初始化 → 滚动触发GSAP动画序列
  ├─ 向下滚动 → 触发历史背景动画 → 出现时间线 → 展示关键节点
  ├─ 继续滚动 → 工程参数浮入 → 数据可视化 → 社会效益展示
  └─ 页面底部 → 导航菜单 → 跳转详细页/记忆页/留言页
```

### 1.4 技术栈选型
| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| 3D渲染 | Three.js r128 | WebGL封装完善，低门槛实现3D粒子效果 |
| 动画 | GSAP 3 + ScrollTrigger | 业界最高性能JS动画库，精确控制帧时序 |
| 布局 | CSS Grid + Flexbox | 响应式布局，PC/平板/手机三端适配 |
| 字体 | Noto Serif SC | 中文衬线字体，契合运河文化主题 |
| 图标 | Font Awesome 6 | 统一视觉语言，减少图片依赖 |

### 1.5 创新点
1. **数字叙事范式**：将传统线下展览转化为Web端沉浸式体验，打破时空限制
2. **3D粒子水流模拟**：使用Three.js在浏览器端实现低GPU开销的粒子效果
3. **滚动驱动动画编排**：GSAP ScrollTrigger将页面滚动位置与动画帧精确绑定
4. **中文Web排版的工程化实践**：字重、行距、字距的系统性调校

## 2. 车牌识别系统 v8.1

### 2.1 项目背景与意义
车牌识别（License Plate Recognition, LPR）是智能交通系统的核心技术之一，广泛应用于停车场管理、违章检测、ETC收费等场景。本项目从教育与实践角度出发，设计并实现了一个完整的LPR教学系统。系统的核心价值在于：(1) 零基础可操作的GUI界面；(2) 集成CNN深度学习、SVM机器学习、EasyOCR三种完全不同的技术路线；(3) 提供量化的准确率对比评估。

### 2.2 系统架构
```
┌──────────────────────────────────────────────────────┐
│                   GUI层 (Tkinter)                     │
│  创建目录 → 生成合成数据 → 选择车牌图像 → 处理图像    │
│  → 手动标注字符 → 训练CNN/SVM → 方法对比 → 性能评估   │
├──────────────────────────────────────────────────────┤
│                   处理引擎层                          │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │
│  │图像预处理    │ │车牌定位      │ │字符分割v7.0  │ │
│  │灰度/二值/    │ │颜色特征/     │ │经典连通域    │ │
│  │边缘检测      │ │轮廓分析      │ │算法          │ │
│  └──────────────┘ └──────────────┘ └──────────────┘ │
├──────────────────────────────────────────────────────┤
│                   识别引擎层                          │
│  ┌──────────┐ ┌──────────────┐ ┌──────────────────┐ │
│  │CNN网络   │ │SVM支持向量机 │ │EasyOCR (可选)    │ │
│  │(PyTorch) │ │(scikit-learn)│ │                  │ │
│  └──────────┘ └──────────────┘ └──────────────────┘ │
├──────────────────────────────────────────────────────┤
│                   数据层                              │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │
│  │合成字符生成器│ │真实车牌标注  │ │模型持久化    │ │
│  └──────────────┘ └──────────────┘ └──────────────┘ │
└──────────────────────────────────────────────────────┘
```

### 2.3 识别流程
```
输入车牌图像 → 灰度化 → 高斯滤波降噪 → Canny边缘检测
→ 颜色特征定位车牌区域 → 透视变换矫正 → Otsu二值化
→ 连通域分析分割字符 → 字符归一化(32×32)
→ [CNN/SVM/EasyOCR三路并行识别] → 投票/置信度融合 → 输出识别结果
```

### 2.4 技术栈选型
| 模块 | 技术选择 | 选型理由 |
|------|---------|---------|
| GUI | Tkinter | Python标准库，零额外依赖，适合教学场景 |
| 图像处理 | OpenCV | 业界标准的CV库，API丰富且文档完善 |
| CNN识别 | PyTorch | 动态计算图，调试友好，适合教学 |
| SVM识别 | scikit-learn | 经典ML库，实现简洁，训练速度快 |
| OCR对比 | EasyOCR | 开箱即用的OCR引擎，作为baseline对比 |
| 模型持久化 | joblib | 轻量级序列化，适合小模型 |

### 2.5 性能指标
- 蓝牌/黄牌颜色定位准确率：90%+
- 字符分割准确率：90%+
- CNN识别准确率：依赖训练数据量
- SVM在小样本下有优势，CNN在大样本下优势明显

### 2.6 创新点
1. **三方法对比框架**：首次在同一系统中集成CNN/SVM/EasyOCR三种技术路线进行量化对比
2. **零外部数据依赖**：内建合成数据生成器，无需任何外部数据集
3. **颜色特征定位策略**：针对蓝黄车牌的颜色空间分析，在特定场景下比边缘检测更鲁棒

## 3. 小怪兽帮帮 · 多智能体桌面助手

### 3.1 项目背景与意义
随着大语言模型（LLM）能力的快速增强，AI不再局限于文本对话，开始具备操控真实数字世界的能力。本项目探索了LLM Agent在Windows桌面操作系统层面的应用可能性——将自然语言指令转化为实际的系统操作（鼠标/键盘/窗口控制），实现"AI操控电脑"的目标。采用LangGraph多智能体协作架构，将复杂的桌面任务拆解为多个子任务，由不同Agent并行/串行执行，显著提升了任务成功率和用户体验。

### 3.2 系统架构
```

│                      交互层                              │
│  ┌──────────┐  ┌─────────────┐  ┌───────────────────┐  │
│  │Web Chat  │  │PyQt6桌面组件│  │语音唤醒 "HeyJarvis"│  │

│  └───────────────────────────────────────────────────┘  │

│  │  │PC控制 │ │微信   │ │系统   │ │搜索   │ ...    │  │

│  └────────────────┘  └──────────────┘  └────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 3.3 任务执行流程
```
用户输入/语音 → STT转文字 → 意图路由(语义匹配)
  ├─ 匹配到PC控制意图 → PC Agent接管
  │    └─ 解析操作目标 → 调用pyautogui/pywin32 → 返回执行结果
  ├─ 匹配到微信意图 → WeChat Agent接管
  │    └─ 解析联系人/消息内容 → 调用微信自动化API → 返回状态
  ├─ 匹配到系统控制意图 → System Agent接管
  │    └─ 解析操作类型 → 调节音量/亮度 → 返回状态
  └─ 匹配到搜索意图 → Search Agent接管
       └─ 构造搜索查询 → DuckDuckGo API → 摘要 + RAG检索
→ 所有结果汇总到主Agent → 自然语言回复用户
```

### 3.4 技术栈选型
| 模块 | 技术选择 | 选型理由 |
|------|---------|---------|
| Agent编排 | LangGraph | 支持复杂的有向图Agent工作流 |
| 意图路由 | Sentence-Transformers | 语义相似度比关键词匹配更鲁棒 |
| 本地LLM | Ollama + Qwen | 隐私优先，离线可用 |
| 云端LLM | Groq | 极低延迟的API推理 |
| 桌面控制 | PyQt6 + pyautogui | 跨Windows版本稳定的键鼠模拟 |
| API代理 | Cloudflare Worker | 解决国内网络访问限制 |

### 3.5 创新点
1. **多Agent协作**：LangGraph编排的Agent图，支持并行/串行/条件分支
2. **桌面级Agent落地**：LLM真正操控操作系统，而非简单的文本交互
3. **双模型热切换**：本地Ollama（隐私）与云端Groq（性能）无缝切换
4. **语音唤醒交互**："Hey Jarvis"关键词检测，免提交互

## 5. Film Genome System · 影视基因组系统

### 5.1 项目背景与意义
传统的AI视频生成（如Sora、Runway）依赖端到端生成模型，缺乏对影视创作规律的显式建模。FGS提出了"检索增强创作"（Retrieval-Augmented Creation, RAC）的全新范式——系统自动分析海量影视片段，提取视觉、音频、叙事、转场四个维度的"创作基因"，存入Qdrant向量数据库。在生成阶段，根据用户需求检索最相关的创作基因，指导LLM生成剧本和分镜，最终合成30-60秒AI短片。这一方法论将"参考→分析→创作"的专业影视工作流数字化。

### 5.2 系统架构
```
┌──────────────────────────────────────────────────────────┐
│                      输入层                               │
│  ┌──────────────────┐  ┌──────────────────────────────┐  │
│  │影视片段上传      │  │参考剧本 / 创作需求           │  │
│  └──────────────────┘  └──────────────────────────────┘  │
├──────────────────────────────────────────────────────────┤
│                      分析引擎层                           │
│  ┌──────────────┐ ┌────────────┐ ┌─────────┐ ┌───────┐  │
│  │视觉DNA分析   │ │音频DNA分析 │ │叙事DNA  │ │转场DNA│  │
```

## 6. BGF · 光伏板缺陷智能检测平台

### 6.1 项目背景与意义
大型光伏电站通常占地数千亩，包含数十万块光伏板。传统的人工巡检效率极低（每人每天仅能巡检约1MW），且难以发现热斑、隐裂等内部缺陷。BGF平台利用"无人机巡检→AI自动检测→GIS精确定位→工单闭环处理"的技术路线，实现了光伏电站运维的全面数字化和智能化。

```
  │  ├─ 化妆师模式 → 预约日程            │

  │  └─ 自习模式 → 专注计时              │
  └──────────────────────────────────────┘
```
